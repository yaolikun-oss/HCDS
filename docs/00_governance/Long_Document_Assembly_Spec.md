# HCDS Long Documentation Assembly Specification

This specification defines how long HCDS documentation is collaboratively generated by ChatGPT and assembled by Codex.

---

## Purpose

Some HCDS specification documents exceed the maximum reliable output length of a single LLM response.

To maintain documentation quality while avoiding context truncation, long documents are generated in multiple sequential parts and automatically assembled by Codex into one final Markdown document.

The final repository must contain **only one complete document**.

Intermediate parts are temporary generation artifacts and must never remain in the repository.

---

## Responsibilities

### ChatGPT

Responsible for:

* writing document content
* preserving document continuity
* maintaining section numbering
* avoiding duplicated sections
* following HCDS documentation standards

ChatGPT outputs the document sequentially.

Example:

```text
Part 1
Sections 1-5

Part 2
Sections 6-10

Part 3
Sections 11-15

Part 4
Sections 16-20
```

Each part continues directly from the previous part.

No section may be rewritten in later parts.

---

### Codex

Responsible for:

* collecting all generated parts
* preserving original order
* merging into one Markdown document
* removing duplicate titles
* removing duplicate headers
* validating numbering
* validating Markdown formatting
* ensuring a single Table of Contents, if present
* writing the final file into the repository

Codex must not rewrite technical content unless explicitly instructed.

---

## Assembly Rules

The final Markdown document must contain exactly one:

* title
* document header
* version block
* status block
* responsibility statement

These appear only once at the beginning of the document.

Subsequent generated parts begin immediately with the next section.

---

## Section Numbering

Section numbering is continuous.

Example:

```text
1
2
3
4
5
6
7
...
20
```

Restarting numbering inside later parts is forbidden.

---

## Cross References

Later parts may reference previous sections.

Example:

```text
See Section 4.

As defined in Section 8.
```

Codex must preserve these references.

---

## Markdown Integrity

During assembly, Codex must verify:

* heading hierarchy
* fenced code blocks
* tables
* lists
* diagrams
* indentation
* blank lines
* anchors
* internal links

No Markdown structure may become invalid after merging.

---

## Forbidden Behavior

Codex must not:

* reorder sections
* summarize content
* rewrite wording
* merge unrelated sections
* delete normative requirements
* renumber headings unless necessary to repair continuity
* invent missing content

---

## Final Output

The repository must contain only the final complete document at its official path.

Examples:

```text
docs/02_architecture/Data_Flow_Architecture.md
docs/03_data_standard/HCDS_Schema.md
docs/03_data_standard/Dataset_Master_Spec.md
docs/05_implementation_standard/LoRA_Training_Spec.md
```

The document must appear as if it had been written in a single continuous editing session.

Readers must not be able to determine that it was generated in multiple parts.

---

## Quality Requirement

The assembled document must satisfy:

* Claude Code implementation-ready
* Codex implementation-ready
* machine-readable
* HCDS documentation standard
* One File = One Responsibility

No manual editing should be required after assembly.

End of specification.
